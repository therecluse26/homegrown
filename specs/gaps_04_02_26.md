## 1 · Critical (P0) — Legal / Security / Data Loss

### CRIT-1 · NoopThornAdapter — no CSAM detection or NCMEC reporting (CARRIED FORWARD)

`internal/safety/noop_adapters.go` — `NoopThornAdapter{}` remains:

- `ScanCsam` always returns `&CsamScanResult{IsCSAM: false}` — no content is ever flagged.
- `SubmitNcmecReport` returns `nil, nil` — no report is ever filed.
- `CheckHashUpdate` returns `false, nil` — hash database never refreshed.

`safety.LoggingThornAdapter{}` is used at runtime (`cmd/server/main.go`), which logs
warnings but still does not perform real scans.

**Impact:** Federal legal obligation under 18 U.S.C. § 2258A requires electronic service
providers to report CSAM to NCMEC. A noop adapter means the platform cannot comply.

**Carried forward from:** core-gaps.md CRIT-2 → gaps_03_31_26.md CRIT-1 (unchanged).

## 2 · High (P1) — Runtime Broken / Silent Failures

### P1-1 · Mkt ContentFlaggedHandler does not auto-archive flagged listings

`internal/mkt/event_handlers.go:50-70` — `ContentFlaggedHandler` logs a warning but does
not call `ArchiveListingByContentKey`. Flagged content remains publicly visible in the
marketplace after safety moderation.

```
slog.Warn("mkt: ArchiveListingByContentKey not yet implemented — flagged listing not auto-archived", ...)
```

**Impact:** Content flagged by the safety domain stays live in the marketplace. Parents may
see moderated content in search results and listings. Combined with CRIT-1 (noop safety
scanner), this creates a double gap — content is neither detected nor removed.

---

### P1-2 · Billing aggregate_payouts job is a no-op

`internal/billing/tasks.go:28-33` — The `billing:aggregate_payouts` handler logs a warning
and returns nil. Creator earnings are never aggregated into `bill_payouts` rows, which means
`execute_payouts` (which *is* implemented) has nothing to process.

**Impact:** Marketplace creators will never receive payouts. The `execute_payouts` handler
correctly processes pending rows via `adapter.CreatePayout()`, but no rows are ever created
because aggregation is a no-op.

---

### P1-3 · Notify HandleFamilyDeletionScheduled is a no-op

`internal/notify/service.go:719` — `HandleFamilyDeletionScheduled` returns `nil` immediately
without creating any notification.

**Impact:** When a family requests account deletion, they receive no confirmation notification.
GDPR Article 17 and COPPA §312.10 best practices recommend explicit notification to the user
that deletion has been initiated. Silent deletion is a UX and compliance concern.

---

### P1-4 · Lifecycle ExportHandlers list is empty

`cmd/server/main.go:1172` — `[]lifecycle.ExportHandler{}` is passed to the lifecycle service.
Data export requests are accepted via the API but produce empty export files because no
domain adapter contributes data.

**Impact:** GDPR Article 20 (data portability) requires that families can export their data.
The export job runs but generates nothing. DeletionHandlers *are* populated (9 adapters), so
deletion works — but export does not.

## 3 · Medium (P2) — Feature Incomplete / Quality

### P2-1 · Search suggestions endpoint returns empty list

`internal/search/handler.go:143-146` — The `suggestions` handler has:
```go
// TODO(phase-3): wire Typesense suggestions query. Returning empty list for now.
return c.JSON(http.StatusOK, map[string]any{"suggestions": []any{}})
```

The autocomplete repository methods (`AutocompleteMarketplace`, `AutocompleteSocial`,
`AutocompleteLearning`) exist in the service layer but are only called from the main
`Search()` method, not from the `suggestions` endpoint.

**Fix:** Wire the `suggestions` handler to call `svc.Autocomplete()` instead of returning
an empty list. This is a Phase 1 fix (PostgreSQL-based), not blocked on Typesense.

### P2-8 · Hand-written API types in frontend hooks

Three hook files define hand-written TypeScript types instead of importing from
`@/api/generated/schema`:

| File | Hand-written interfaces |
|------|:----------------------:|
| `frontend/src/hooks/use-recommendations.ts` | 2 |
| `frontend/src/hooks/use-search.ts` | 16 |
| `frontend/src/hooks/use-subscription.ts` | 6 |

**Root cause:** Backend swag annotations for recs, search, and billing do not produce
response shapes matching what the frontend consumes. Running `make full-generate` does not
produce these types in `schema.ts`.

**Fix:** Add/correct swag annotations on recs, search, and billing handlers, then replace
hand-written interfaces with generated imports.

**Carried forward from:** gaps_03_31_26.md FE-6 (unchanged).
