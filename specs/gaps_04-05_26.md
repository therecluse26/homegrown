## 1 · Critical (P0) — Legal / Security / Data Loss

### CRIT-1 · NoopThornAdapter — no CSAM detection or NCMEC reporting (CARRIED FORWARD)

`internal/safety/noop_adapters.go` — `NoopThornAdapter{}` remains:

- `ScanCsam` always returns `&CsamScanResult{IsCSAM: false}` — no content is ever flagged.
- `SubmitNcmecReport` returns `nil, nil` — no report is ever filed.
- `CheckHashUpdate` returns `false, nil` — hash database never refreshed.

`safety.LoggingThornAdapter{}` is used at runtime (`cmd/server/main.go`), which logs
warnings but still does not perform real scans. `ManualReviewThornAdapter` was added in the
previous cycle but only queues items for human review — it does not perform automated
scanning or hash matching.

**Impact:** Federal legal obligation under 18 U.S.C. § 2258A requires electronic service
providers to report CSAM to NCMEC. No adapter performs real detection.

**Carried forward from:** core-gaps.md CRIT-2 → gaps_03_31_26.md CRIT-1 →
gaps_04_02_26.md CRIT-1 (unchanged).

---

### CRIT-2 · CSRF protection not implemented

`internal/app/app.go` and `internal/middleware/` — CODING_STANDARDS §4 requires CSRF
protection for all state-mutating endpoints. No CSRF middleware exists anywhere in the
middleware stack.

The Echo framework supports CSRF middleware (`echo.middleware.CSRF`), and the app configures
other security middleware (CORS, security headers, rate limiting), but CSRF token generation
and validation are absent.

**Impact:** State-mutating requests (POST/PUT/PATCH/DELETE) from authenticated sessions are
vulnerable to cross-site request forgery. An attacker-controlled page could trigger actions
(e.g., `POST /families/co-parents/invite`, `DELETE /families`) using a victim's session
cookies.

---

### CRIT-3 · WebSocket stale session — no re-validation after connect

`internal/social/websocket.go:53-56` — The WebSocket upgrade handler validates the session
once at connection time but never re-checks session validity during the connection lifetime.

If a parent's session is revoked (e.g., by a co-parent removal, account suspension, or
logout-all), the WebSocket connection remains open indefinitely. The revoked parent
continues to receive real-time chat messages, notifications, and presence updates.

**Impact:** Revoked or suspended users retain real-time data access until they manually
close the browser tab or the connection drops for network reasons.

## 2 · High (P1) — Runtime Broken / Silent Failures

### P1-1 · Billing webhook errors silently return 200

`internal/billing/service.go:566-714` — The billing webhook handler catches processing
errors internally but returns HTTP 200 to the payment provider (Hyperswitch) regardless of
outcome. This prevents the provider from retrying failed webhooks.

**Impact:** Failed payment state transitions (subscription activated, payment failed, refund
processed) are silently lost. The platform's billing state drifts from the provider's state,
leading to incorrect subscription statuses, missed cancellations, and un-applied refunds.

---

### P1-2 · `UpdateStudentMethodology` returns "not_implemented"

`internal/method/service.go:248` — The `UpdateStudentMethodology` method returns a
hard-coded `"not_implemented"` error. The endpoint is registered, accepts requests, but
always fails.

**Impact:** Parents cannot change a student's methodology after initial setup. The
onboarding wizard and settings UI may call this endpoint expecting it to work, resulting in
a confusing error.

---

### P1-3 · IAM Phase 2 co-parent methods panic("not implemented")

`internal/iam/handler_test.go:105-193` — Test file documents that co-parent Phase 2
methods (`transferPrimaryParent`, `withdrawCoppaConsent`) contain `panic("not implemented")`
in the service layer. If these endpoints are hit in production, the server crashes.

**Impact:** Server process crash on any call to transfer primary parent or withdraw COPPA
consent. While these may be uncommon flows, a panic in production is always P1.

---

### P1-4 · Notify email batching deferred (sends per-family)

`internal/notify/service.go:484` — Email delivery sends one email per family per
notification event. For broadcast notifications (e.g., platform announcements, safety
alerts), this means N individual SMTP/SES calls instead of batched delivery.

**Impact:** At scale, notification broadcasts will hit email provider rate limits and incur
unnecessary latency. For a platform with 10K+ families, a single announcement triggers 10K
individual API calls. Not broken at current scale, but a ticking time bomb.

---

### P1-5 · Learn tool resolution cache invalidation missing

`internal/learn/service.go:2579` — The tool resolution system caches resolved tools
(curriculum resources mapped to methodology-specific implementations) but has no cache
invalidation when underlying tool definitions change.

**Impact:** When an admin updates tool definitions or a marketplace listing changes, stale
cached resolutions persist until server restart. Families may see outdated or incorrect
tool recommendations.

---

### P1-6 · Manual review adapter silently discards non-CSAM scans

`internal/safety/manual_review_adapter.go:14,63` — The `ManualReviewThornAdapter` only
processes CSAM-flagged content (items where `IsCSAM: true`). Content that passes the
initial scan but has other safety concerns (e.g., age-inappropriate material, violence)
is silently discarded without queuing for review.

**Impact:** The manual review queue only contains definitive CSAM positives. Borderline or
non-CSAM safety concerns (which are far more common) bypass the review pipeline entirely.

## 3 · Medium (P2) — Correctness / Observability

### P2-1 · Zero metrics/observability (no Prometheus/OTel)

The entire codebase has no metrics instrumentation. No Prometheus metrics, no OpenTelemetry
traces, no structured performance counters. The only observability is `slog` logging.

**Impact:** No alerting on error rates, latency degradation, or resource exhaustion.
Incidents can only be detected by user reports or manual log inspection. Standard for
production Go services is at minimum a `/metrics` Prometheus endpoint.

---

### P2-2 · Health endpoint doesn't check DB/Redis/Kratos

`internal/app/app.go:128` — The health endpoint returns a static 200 OK without checking
the health of downstream dependencies (PostgreSQL, Redis, Kratos, Asynq).

**Impact:** Load balancers and orchestrators (Kubernetes, etc.) cannot detect when the
service is running but unable to serve requests due to a database outage, Redis failure,
or Kratos unavailability. The pod stays in rotation, accepting requests it cannot fulfill.

---

### P2-3 · 7 handler files have zero swagger annotations

The following handler files have no `@Summary`, `@Router`, or other swag annotations,
meaning their endpoints are invisible in the generated OpenAPI spec and TypeScript types:

| Domain | Handler file |
|--------|-------------|
| 16-admin | `internal/admin/handler.go` |
| 06-learn | `internal/learn/handler.go` |
| 15-lifecycle | `internal/lifecycle/handler.go` |
| 07-mkt | `internal/mkt/handler.go` |
| 09-media | `internal/media/handler.go` |
| 08-notify | `internal/notify/handler.go` |
| 11-safety | `internal/safety/handler.go` |

**Impact:** Frontend cannot use generated types for these domains — hand-written types or
`any` casts are required, violating CODING_STANDARDS zero-any rule.

### P2-4 · No retry/circuit-breaker on external calls (Kratos, Hyperswitch, Typesense)

Multiple service files make HTTP calls to external services without retry logic or circuit
breakers:

- IAM → Kratos admin API (identity management)
- Billing → Hyperswitch (payment processing)
- Search → Typesense (full-text search)
- Safety → Thorn API (content scanning)

**Impact:** Transient failures (network blips, service restarts) cause immediate user-facing
errors. A brief Kratos outage breaks all authentication flows. No exponential backoff or
circuit-breaking to prevent cascading failures.

---

### P2-5 · `return nil, nil` pattern conflates "not found" with "error"

Multiple repository methods across comply, billing, learn, and mkt domains return
`nil, nil` when a record is not found, making it impossible for callers to distinguish
between "record doesn't exist" and "query succeeded with no error."

**Impact:** Callers must add nil-checks on both the result and error, and "not found" cannot
be reliably propagated as a 404 HTTP response. Some callers may treat `nil, nil` as success
when they should return a not-found error.

---

### P2-6 · Student token endpoint lacks rate limiting

`internal/iam/service.go:903-920` — The student session token generation endpoint has no
rate limiting. Student tokens are simplified auth tokens (no Kratos session), making them
easier to brute-force.

**Impact:** An attacker could enumerate student session tokens by repeatedly hitting the
endpoint. While tokens are UUIDs (high entropy), the lack of rate limiting is a defense-
in-depth gap.

---

### P2-7 · Dynamic SQL column names in comply repo (future injection risk)

`internal/comply/repository.go` — The comply repository constructs SQL queries with dynamic
column names derived from function parameters. While the current callers pass safe values,
the pattern is fragile — a future caller could pass user-supplied input.

**Impact:** Not exploitable today (callers use hardcoded strings), but the pattern violates
defense-in-depth. A future developer adding a new caller might inadvertently pass
unsanitized input.

**Fix:** Use an allowlist of valid column names and validate before interpolation.

---

### P2-8 · Pagination params have no upper-bound validation

All handler files accept `limit` and `offset` query parameters without upper-bound
validation. A client can request `?limit=1000000`, causing the database to return and
serialize massive result sets.

**Impact:** Memory exhaustion and query timeout DoS vector. A malicious or buggy client
can degrade service for all users by requesting unbounded pages.

**Fix:** Add a `MaxPageSize` constant (e.g., 100) in shared and clamp `limit` in all
handlers.

---

### P2-9 · Feature flags exist in admin but no runtime consumer API

`internal/admin/` — The admin domain has a full feature flag CRUD system (create, update,
delete, list flags with targeting rules). However, no runtime API exists for other domains
or the frontend to evaluate flags.

**Impact:** Feature flags can be defined but never consumed. Domains cannot conditionally
enable features, and the frontend cannot check flag state. The feature flag system is
write-only.

---

### P2-10 · Missing security headers (Permissions-Policy, COOP, CORP)

`internal/middleware/security_headers.go` — The security headers middleware sets some
standard headers but is missing:

- `Permissions-Policy` — controls browser feature access (camera, microphone, geolocation)
- `Cross-Origin-Opener-Policy` (COOP) — prevents cross-origin window references
- `Cross-Origin-Resource-Policy` (CORP) — controls cross-origin resource loading

**Impact:** The platform does not restrict browser feature access or cross-origin isolation.
While not immediately exploitable, these headers are part of modern defense-in-depth and are
expected by security auditors.

Integration test files in discover, iam, method, and onboard domains use `t.Skip()` when
a test database is unavailable. This means CI environments without a PostgreSQL instance
silently skip all integration tests.

**Impact:** Integration test regressions go undetected in CI. The tests exist and pass

implemented.

**Impact:** Users cannot add payment methods through the UI. The subscription upgrade flow
may be broken if it depends on having a payment method on file.

### P3-5 · Creator verification tax ID placeholder hardcoded

`frontend/src/features/settings/creator-verification.tsx` — The tax ID field uses a
hardcoded placeholder format (e.g., "XXX-XX-XXXX") rather than adapting to the user's
locale or country.

**Impact:** Minor UX issue — international users see US-formatted tax ID hints. No
functional impact.

---

### P3-6 · CoParent invite token comparison timing-vulnerable

`internal/iam/service.go:628-632` — The co-parent invite acceptance flow compares the
provided token to the stored token using a direct string comparison (`==`). This is
vulnerable to timing side-channel attacks.

**Impact:** Low practical risk (tokens are UUIDs with high entropy, and the attack requires
high-precision timing over a network), but best practice is to use `crypto/subtle.ConstantTimeCompare`.

---

### P3-7 · Undocumented env vars (OBJECT_STORAGE_PUBLIC_URL, UNSUBSCRIBE_SECRET)

`config.go` references environment variables that are not documented in any README,
`.env.example`, or deployment guide:

- `OBJECT_STORAGE_PUBLIC_URL` — public base URL for uploaded files
- `UNSUBSCRIBE_SECRET` — HMAC key for unsubscribe link generation

**Impact:** New developers or deployment automation may miss required configuration, leading
to runtime failures or insecure defaults.

---

### P3-8 · Webhook secrets validated at first use, not startup

`config.go` — Webhook signing secrets (Hyperswitch, Kratos) are loaded from environment
variables but not validated at startup. If a secret is missing or malformed, the error
surfaces only when the first webhook arrives.

**Impact:** Deployment misconfiguration is not detected until runtime. A missing webhook
secret could cause silent webhook rejection, leading to the issues described in P1-1.

## 7 · Prioritized Remediation Roadmap

### P0 — Legal / Security (before any public beta)

1. **CRIT-1: Real CSAM detection** — Integrate Thorn Safer / PhotoDNA for automated hash
   matching. `ManualReviewThornAdapter` is insufficient alone; detection must be automated.
2. **CRIT-2: CSRF middleware** — Add Echo CSRF middleware to all state-mutating routes.
   Exempt webhook endpoints (they use signature verification) and API-key-authenticated
   routes.
3. **CRIT-3: WebSocket session re-validation** — Implement periodic session checks (e.g.,
   every 60s) on active WebSocket connections. Disconnect on invalid/revoked session.

### P1 — Runtime Correctness (before beta users)

4. **P1-1: Billing webhook error propagation** — Return appropriate HTTP status codes (4xx/5xx)
   on processing failure so the payment provider retries.
5. **P1-2: Implement UpdateStudentMethodology** — Remove `not_implemented` stub; implement
   methodology reassignment with config migration.
6. **P1-3: Replace panic with error returns** — `transferPrimaryParent` and
   `withdrawCoppaConsent` must return errors, not panic.
7. **P1-5: Cache invalidation** — Add TTL or event-driven invalidation for learn tool
   resolution cache.
8. **P1-6: Manual review adapter scope** — Queue all safety-flagged content for review,
   not just CSAM-positive items.
9. **P1-4: Email batching** — Implement `SendBatch()` in the email adapter for broadcast
   notifications. Lower priority than other P1s (only affects scale).

### P2 — Correctness / Observability (before GA)

10. **P2-1: Prometheus metrics** — Add `/metrics` endpoint with request latency, error rates,
    DB pool stats, queue depths.
11. **P2-2: Deep health checks** — Health endpoint should verify DB, Redis, Kratos, Asynq
    connectivity.
12. **P2-3: Swagger annotations** — Add swag annotations to 7 handler files; regenerate types.
13. **P2-8: Pagination bounds** — Add `MaxPageSize` validation in shared; apply to all handlers.
14. **P2-10: Security headers** — Add Permissions-Policy, COOP, CORP to security headers
    middleware.
15. **P2-5: Eliminate nil,nil** — Return `shared.ErrNotFound` instead of `nil, nil` in repos.
16. **P2-6: Rate limit student tokens** — Apply per-IP rate limiting to student token endpoint.
17. **P2-7: SQL column allowlist** — Validate dynamic column names against allowlist in comply.
18. **P2-4: Retry/circuit-breaker** — Add retry with exponential backoff for Kratos,
    Hyperswitch, Typesense calls. Consider `sony/gobreaker` for circuit-breaking.
19. **P2-9: Feature flag evaluation API** — Add `GET /feature-flags/evaluate` endpoint for
    runtime flag checks.

### P3 — Polish / Future (post-GA)

20. **P3-6: Constant-time token comparison** — Use `crypto/subtle.ConstantTimeCompare` for
    co-parent invite tokens.
21. **P3-8: Startup secret validation** — Validate all webhook secrets at startup, fail fast.
22. **P3-7: Document env vars** — Add `.env.example` with all required/optional variables.
23. **P3-1: Generated frontend types** — Fix remaining swag annotations; replace hand-written
    interfaces.
24. **P3-2: a11y in CI** — Add a11y tests to CI pipeline (requires headless browser setup).
25. **P3-3: Integration test DB in CI** — Provision test PostgreSQL in CI for integration tests.
26. **P3-4: Payment method form** — Implement Hyperswitch payment method collection UI.
27. **P3-5: Locale-aware tax ID** — Adapt placeholder format based on user locale.
