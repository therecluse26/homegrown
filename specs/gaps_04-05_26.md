## 2  Gap Register

### 2.1  Critical

#### GAP-C1 — No Periodic Retention Enforcement Job

| Field | Value |
|-------|-------|
| **Spec ref** | SPEC §14.4, ARCH §2.14 |
| **Current state** | `ProcessDeletion()` exists in `internal/lifecycle/service.go:329` and handles deletion orchestration correctly. However, it is **not registered as a periodic job** in the scheduler (`cmd/server/main.go:1490`). Deletions are only triggered on-demand (COPPA immediate) or via the `FamilyDeletionScheduled` IAM event. Grace-period expirations (`pending` → `processing`) have no automatic sweep. |
| **Risk** | Families who request deletion and wait out the 30-day grace period will never have their data actually removed unless an admin manually triggers processing. Violates GDPR Art. 17 and COPPA §312.10 timelines. |
| **Effort** | S — Register `lifecycle:process_deletion` in `JobScheduler.Register()` with a daily cron expression (e.g., `0 2 * * *`). The handler already exists in `internal/lifecycle/tasks.go:30`. |

#### GAP-C2 — Cloud Storage Adapter Not Implemented

| Field | Value |
|-------|-------|
| **Spec ref** | SPEC §9.7, ARCH §2.10 |
| **Current state** | `ObjectStorageAdapter` interface defined in `internal/media/ports.go:131`. Only implementation is `NoopStorageAdapter` in `internal/media/noop_adapters.go` (returns placeholder URLs, logs warnings). `config.go:107` documents the optional S3 fields but no real adapter exists. |
| **Risk** | All file uploads (student work, media, portfolio evidence) are silently discarded in production. Media domain is non-functional without a real storage backend. |
| **Effort** | M — Implement S3-compatible adapter (R2/MinIO) using `aws-sdk-go-v2`. Interface has 8 methods; all have clear signatures. |

#### GAP-H3 — External CSAM Scanning Uses Interim Manual Review

| Field | Value |
|-------|-------|
| **Spec ref** | SPEC §11.6, 18 U.S.C. §2258A |
| **Current state** | `internal/safety/adapters/manual_review_adapter.go` returns `RequiresManualReview=true` for all uploads. Full CSAM detection pipeline exists: detection → quarantine → NCMEC report → account ban (non-appealable). `SubmitNcmecReport` persists reports to `safety_ncmec_pending_reports` for manual filing. | **Risk** | Without real Thorn Safer / PhotoDNA credentials, every upload requires manual review — not scalable. NCMEC reports must be filed manually rather than automatically. Legally compliant (18 U.S.C. §2258A only requires reporting *known* CSAM) but operationally fragile. |
| **Effort** | S — Obtain Thorn Safer API credentials, implement `ThornScanAdapter` behind existing `CsamScanner` interface. Pipeline infrastructure is already complete. |

#### GAP-H4 — Minimal Observability Infrastructure

| Field | Value |
|-------|-------|
| **Spec ref** | ARCH §2.15 (implied) |
| **Current state** | `internal/middleware/metrics.go` registers exactly **2 Prometheus metrics**: `http_requests_total` (CounterVec) and `http_request_duration_seconds` (HistogramVec). Metrics endpoint at `GET /metrics`. No domain-specific metrics, no database query metrics, no background job metrics, no distributed tracing. |
| **Risk** | Cannot diagnose production issues effectively. No visibility into: upload processing latency, search query performance, job queue depth, subscription processing time, or cache hit rates. |
| **Effort** | L — Phased rollout: (1) Add domain-specific counters/histograms to critical paths, (2) Add job queue metrics, (3) Integrate OpenTelemetry tracing. |

#### GAP-H5 — Inconsistent Deletion Patterns Across Domains

| Field | Value |
|-------|-------|
| **Spec ref** | SPEC §14, CODING_STANDARDS §6 |
| **Current state** | 9 domains register `DeletionHandler` adapters (learn, social, mkt, media, notify, billing, comply, recs, plan). However, deletion implementation varies: **5 domains** use `DeleteByFamily` repository methods with proper cascade (comply, recs, billing, mkt, notify). **4 domains** use divergent patterns — learn and media use `BypassRLSTransaction` with manual cascade; social returns nil (no-op); plan uses custom `DeleteData`/`DeleteStudentData`. **6 domains** have no deletion support at all (safety, search, method, discover, onboard, iam). |
| **Risk** | Incomplete data deletion violates GDPR right-to-erasure. The 6 unregistered domains may store family-scoped data that survives deletion requests. |
| **Effort** | M — Audit each unregistered domain for family-scoped data; implement `DeletionHandler` where needed; standardize on `DeleteByFamily` pattern. |

| Field | Value |
|-------|-------|
| **Spec ref** | ARCH §2.14 |
| **Current state** | Background jobs use asynq (Redis-backed). Job types: `lifecycle:data_export`, `lifecycle:process_deletion`, `safety:csam_report`, `safety:check_csam_hash_update`, `learn:snapshot_progress`. No dashboard, no dead-letter queue alerts, no job duration metrics. |

#### GAP-M4 — Missing `DeleteByFamily` in Domains with Family-Scoped Data

| Field | Value |
|-------|-------|
| **Spec ref** | SPEC §14, CODING_STANDARDS §3.2 |
| **Current state** | Subset of GAP-H5. Domains `learn`, `social`, `media`, `plan` implement deletion through DeletionHandler adapters but lack standardized `DeleteByFamily` repository methods. The adapters work via `HandleFamilyDeletionScheduled` event handlers or custom methods. |
| **Risk** | Inconsistent deletion logic is harder to audit for completeness. If a new table is added to these domains, the custom deletion path may miss it. |
| **Effort** | M — Refactor to use `DeleteByFamily` repository methods called from DeletionHandler adapters, matching the pattern in comply/recs/billing/mkt/notify. |

## 6  Methodology Note

This audit was conducted by:

1. **Reflex MCP** code search across all 18 domains for interface implementations,
   handler registrations, adapter patterns, and test coverage
2. **Manual review** of `cmd/server/main.go` wiring (lines 1–2400) to verify all
   domain registrations, adapter bindings, and job scheduler configuration
3. **Cross-reference** against `specs/SPEC.md` section requirements and
   `specs/ARCHITECTURE.md` ADRs for each identified gap

Previous gap analyses (now deleted) over-estimated the number of critical gaps by
classifying fully-implemented domains as "stubbed." This report corrects those
assessments based on actual code inspection.
